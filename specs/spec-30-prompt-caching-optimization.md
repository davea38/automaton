# Spec 30: Prompt Caching Optimization

## Purpose

Prompt cache reads cost 90% less than uncached input tokens (0.1x base price). Automaton does not currently design for cache hits — timestamps, iteration numbers, and budget figures are injected into the prompt prefix, invalidating the cache every iteration. This spec defines prompt structure rules, cache hit ratio tracking, and parallel builder cache sharing to maximize cache efficiency.

## Requirements

### 1. Static Prefix / Dynamic Suffix Prompt Structure

Every prompt assembled by the orchestrator must follow this structure:

```
┌──────────────────────────────────────┐
│  STATIC PREFIX (identical every run) │  ← cached
│  - System prompt / agent identity    │
│  - CLAUDE.md content                 │
│  - Rules and constraints             │
│  - Phase instructions                │
│  - Project specs (if included)       │
├──────────────────────────────────────┤
│  DYNAMIC SUFFIX (changes per iter)   │  ← uncached
│  - Iteration number                  │
│  - Current task assignment           │
│  - Recent git diffs                  │
│  - Budget remaining                  │
│  - Context summary                   │
│  - Iteration memory                  │
└──────────────────────────────────────┘
```

The static prefix must be byte-for-byte identical across iterations to achieve cache hits. The orchestrator must NOT inject any of the following into the static prefix:
- Timestamps or dates
- Iteration counts or phase numbers
- Budget figures or cost estimates
- Wave numbers or builder IDs
- File diffs or recent changes

### 2. PROMPT_*.md Static/Dynamic Split

Each `PROMPT_*.md` file must be split into two clearly marked sections per spec-29's template:

```markdown
<!-- STATIC CONTENT — do not inject per-iteration data above this line -->
<context>...</context>
<identity>...</identity>
<rules>...</rules>
<instructions>...</instructions>
<output_format>...</output_format>

<!-- DYNAMIC CONTEXT BELOW — injected by orchestrator -->
<dynamic_context>
{{ITERATION_STATE}}
{{RECENT_CHANGES}}
{{BUDGET_STATUS}}
{{TASK_ASSIGNMENT}}
</dynamic_context>
```

The orchestrator reads the static section verbatim and appends the dynamic section with substituted values. The static section is the cache key.

### 3. Cache Hit Ratio as First-Class Metric

Track cache hit ratio after every agent invocation:

```
cache_hit_ratio = cache_read_input_tokens / (cache_read_input_tokens + input_tokens + cache_creation_input_tokens)
```

Fields already parsed by spec-07 (`cache_read_input_tokens`, `cache_creation_input_tokens`, `input_tokens`) provide all necessary data.

Add to `.automaton/budget.json` history entries:
```json
{
  "cache_hit_ratio": 0.72,
  "cache_read": 80000,
  "cache_create": 5000,
  "uncached_input": 27000
}
```

### 4. Cache Hit Ratio Warning

After 3 or more iterations in any phase, if the rolling average cache hit ratio drops below 50%, emit a warning:

```
[ORCHESTRATOR] WARNING: Cache hit ratio 34% (last 3 iterations). Static prefix may be changing between iterations. Check prompt assembly.
```

This indicates the static prefix is not stable — likely a bug in prompt assembly injecting dynamic data into the prefix.

### 5. Parallel Builder Cache Sharing

When `parallel.enabled` is `true`, all builders in a wave must share an identical static prompt prefix. This ensures the cache entry created by builder-1 is reused by builders 2..N.

Requirements:
- The builder wrapper (spec-17) must construct the task-specific header as part of the DYNAMIC suffix, not the static prefix
- Builder number, wave number, task assignment, and file ownership list go in `<dynamic_context>`, not in the prompt header
- The shared static prefix includes: `PROMPT_build.md` static section + CLAUDE.md + project-level context
- Stagger delay (`parallel.stagger_seconds`) gives the first builder time to populate the cache before others start

### 6. Context Summaries After Static Content

Context summaries generated by spec-24 (`.automaton/context_summary.md`) and iteration memory (`.automaton/iteration_memory.md`) are dynamic content — they change every iteration. They must be placed in the `<dynamic_context>` section, AFTER the static prefix.

Incorrect (breaks cache):
```
[context_summary.md content]    ← dynamic, changes every phase
[PROMPT_build.md static]        ← static, but cache already invalidated
```

Correct:
```
[PROMPT_build.md static]        ← static, cache hit
[context_summary.md content]    ← dynamic, appended after cache boundary
```

### 7. Cache Metrics in Observability

Add cache hit ratio to the dashboard (spec-21) and status line output:

```
# Status line (API mode):
[BUILD 7] auth middleware | ~$2.04 | cache: 72% | budget: $18.60 remaining

# Status line (allowance mode):
[BUILD 7] auth middleware | 112K/24K tokens | cache: 72% | allowance: 32M remaining
```

### 8. Cache and Allowance Interaction

In allowance mode (spec-23), cache read tokens still count against the weekly allowance — the API charges them at the reduced rate but they consume allowance capacity. The budget tracking must account for this:

- `tokens_used_this_week` includes all token types: input, output, cache_create, cache_read
- Cost estimation uses cache pricing for informational display
- High cache hit ratios reduce USD cost but do NOT reduce allowance consumption proportionally (the allowance is token-based, not cost-based)

Document this distinction clearly in budget display when cache ratio is high but allowance consumption is still significant.

### 9. Minimum Cacheable Token Threshold

The static prefix must meet minimum token thresholds for caching to activate:

| Model | Min Tokens |
|-------|-----------|
| Opus 4.6 | 4096 |
| Sonnet 4.6 | 2048 |
| Haiku 4.5 | 4096 |

If the static prefix is below the threshold, the orchestrator should log a one-time informational message per phase:

```
[ORCHESTRATOR] INFO: Static prefix (1800 tokens) below cache threshold for sonnet (2048). Caching inactive.
```

## Acceptance Criteria

- [ ] No dynamic data (timestamps, iteration numbers, budget) appears in the static prefix of any prompt
- [ ] `PROMPT_*.md` files have clear `<!-- STATIC CONTENT -->` / `<!-- DYNAMIC CONTEXT -->` markers
- [ ] Cache hit ratio calculated and stored in budget.json history after every iteration
- [ ] Warning emitted when cache hit ratio < 50% after 3+ iterations
- [ ] All parallel builders share an identical static prompt prefix
- [ ] Context summaries and iteration memory placed after the static prefix
- [ ] Cache hit ratio displayed in status line and dashboard
- [ ] Allowance-mode budget correctly accounts for cache read tokens
